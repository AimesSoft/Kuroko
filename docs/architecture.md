# Kuroko Architecture

Kuroko is an independent Rust-first player. Host frontends call into the
player through the Rust API first, a minimal opaque-handle C ABI now, and
Flutter bindings after that.

Normal playback frames stay inside Kuroko. Video, subtitle, and danmaku
composition are renderer responsibilities and do not flow through Flutter.

## macOS v0 Direction

```text
Rust Player Core
  -> source abstraction: file + HTTP range
  -> FFmpeg custom AVIO probe/demux/decode wrappers
  -> playback session + early video clock + PCM audio blocks
  -> VideoToolbox hardware frame output
  -> CoreAudio output ring buffer
  -> overlay timeline for subtitles and danmaku
  -> renderer core: color state + render graph + tone map/scaler policy
  -> Metal renderer first, wgpu fallback boundary next
  -> presenter runtime
  -> CAMetalLayer presenter
```

The first runnable target is `macos_native_demo`. Without arguments it validates the native Metal
output surface by rendering animated clear colors. With a media path argument it opens the media,
decodes VideoToolbox frames, imports them into Metal, and draws them into the `CAMetalLayer`.
Passing `--smoke-seconds N` keeps the same native path but terminates the AppKit app automatically,
so GUI validation can run as a normal development check.

## Native Dependencies

`xtask` owns native dependency orchestration. It downloads pinned upstream
archives into `third_party/cache`, extracts sources into `third_party/src`, and
installs build products under `third_party/dist/<profile>`.

The default native profile is `lgpl`. `gpl-full` is reserved for later optional
components and is not enabled by default.

Current status:
- FFmpeg 7.1.1 is source-built and linked statically from `third_party/dist/lgpl/ffmpeg`.
- Low-level bindings are generated by `kuroko_ffmpeg_sys` at Cargo build time.
- `kuroko::ffmpeg` provides safe probe, custom-AVIO source input, demux packet, seek,
  software decode, audio resampling, and VideoToolbox hardware decode wrappers.
- libass, HarfBuzz, and FreeType are pinned and fetched on demand with `xtask deps fetch --all`;
  `xtask deps status` reports whether their sources are ready. Their static build/link steps land
  with the libass-backed subtitle renderer.

## Probe Milestone

The `ffmpeg_probe` example validates that the Rust player can open media through FFmpeg and read
container, track, codec, pixel format, and HDR metadata. `ffmpeg_probe_source` validates the same
metadata path through `MediaSource` and FFmpeg custom `AVIOContext` callbacks. The HDR HEVC sample
currently reports HEVC Main 10, `yuv420p10le`, BT.2020 primaries, and PQ transfer, which is the
metadata path the future VideoToolbox and Metal HDR renderer will consume.

## Demux And Decode Milestone

`kuroko::ffmpeg::Demuxer` owns an `AVFormatContext`, optionally with a Rust-owned custom
`AVIOContext` backed by `MediaSource`, exposes stream metadata, supports stream selection, reads
reference-counted packets, and seeks using FFmpeg timestamps. `Packet` owns its `AVPacket` lifetime
and carries stream time base information so decoder queues can pass it across worker threads without
copying compressed payloads.

`kuroko::ffmpeg::Decoder` currently opens FFmpeg software decoders from demuxer codec
parameters and outputs owned `Frame` objects. This is not the final macOS HDR presentation path,
but it proves the packet-to-frame lifecycle and lets us validate frame PTS, dimensions, pixel
format, BT.2020 primaries, and PQ transfer before replacing the backend with VideoToolbox and
CVPixelBuffer/CVMetalTextureCache.

## VideoToolbox Milestone

`DecoderConfig::videotoolbox()` configures FFmpeg with `AV_HWDEVICE_TYPE_VIDEOTOOLBOX`, installs a
hardware pixel-format selector, and outputs owned hardware `Frame` objects. On the HDR HEVC sample,
the frames report `videotoolbox_vld`, carry a hardware frames context, and preserve BT.2020/PQ
metadata. A debug-only transfer path confirms the hardware frame can become `p010le` system memory;
normal playback should avoid that copy and instead bridge the underlying CVPixelBuffer into Metal
through CVMetalTextureCache.

## Metal Import Milestone

`Frame::videotoolbox_pixel_buffer()` exposes the VideoToolbox `CVPixelBufferRef` while the owning
FFmpeg frame is alive. `MetalRenderer::import_video_frame_textures()` creates a `CVMetalTextureCache`
and imports the luma/chroma planes as renderer-owned `CVMetalTexture`/`MTLTexture` objects.

On the HDR HEVC sample, CoreVideo reports `x420` (P010 / 10-bit 4:2:0). The current import path maps
plane 0 to `MTLPixelFormatR16Unorm` at 3840x2160 and plane 1 to `MTLPixelFormatRG16Unorm` at
1920x1080. This is the zero-copy bridge the next Metal shader/compositor milestone will sample from;
no playback frame needs to round-trip through CPU memory.

## Audio Decode Milestone

`kuroko::ffmpeg::AudioResampler` wraps FFmpeg `libswresample` and converts decoded audio frames
into the player-internal PCM format: interleaved f32, currently defaulting to 48 kHz stereo. The HDR
sample's AAC track reports 48 kHz, 6 channels, `fltp`, and is converted to 48 kHz stereo PCM blocks
of 1024 frames. This is the format the first CoreAudio output backend consumes.

`ffmpeg_audio_decode` validates the raw FFmpeg audio path. `playback_audio_tick` validates the same
capability through `PlaybackSession`, so audio decoding is now part of the reusable player kernel
instead of only an isolated example.

## Playback Session Milestone

`kuroko::playback::PlaybackSession` is the first reusable player-kernel layer above the raw
FFmpeg wrappers. It opens media from a `MediaRequest`, chooses default video and audio tracks,
configures the requested video decoder backend, exposes the opened media summary, supports seek,
returns owned decoded video frames, and returns f32 PCM audio blocks. `Player::open` now uses this
session and emits real duration, track, and video parameter events instead of only changing state.

`VideoPlaybackEngine` adds the first clocked playback loop. It drives both video and audio ticks,
with play, pause, stop, seek, media time, pending frames, EOF state, and PTS-based scheduling.
`PlaybackClock` owns the media-time anchor, playback rate, and master-clock source, while
`VideoFrameScheduler` owns the present/wait/drop decision for decoded video frames. The first
audio-master path disciplines the media clock toward decoded audio PTS with deadband, bounded
per-frame correction, and large-drift snap behavior. `AudioRingBuffer` now tracks the PTS of queued
PCM samples and exposes an `AudioClockSnapshot`; the macOS presenter feeds CoreAudio output snapshots
back to the player worker through a non-blocking command so the playback clock can be disciplined by
the actual output queue. `DisplaySyncState` provides the first pure Rust vsync quantizer that carries
residual frame-duration error across frames.

## Overlay Foundation

`kuroko::subtitle` owns early SRT/WebVTT/ASS timeline parsing and can produce debug RGBA
subtitle planes. `SubtitleRendererCore` is the first renderer-facing subtitle boundary: the current
debug timeline path reports changed/unchanged frames, and the feature-gated libass renderer accepts
ASS scripts, calls `ass_render_frame`, and imports libass alpha planes plus RGBA/inverse-alpha color
into Kuroko straight-RGBA overlay planes. `LibassRenderPlan` records the renderer operation order
used by mature ASS paths (`set_frame_size`, `set_storage_size`, cache limits, then `render_frame`),
and `RawAssImage` mirrors the ABI prefix needed to import an `ASS_Image` list.
`kuroko::danmaku` owns Bilibili XML/JSON-lines parsing, timeline windows, lane placement, and
collision-aware layout boxes. `kuroko::overlay::OverlayTimeline` combines debug subtitles or
libass-backed subtitles with danmaku for a timestamp and viewport without routing overlay content
through Flutter.

`MetalRenderer::render_video_frame_with_overlay()` now uploads subtitle RGBA planes into Metal
textures and alpha-blends them over the current video drawable. Danmaku layout boxes are still
prepared as renderer-owned batches; the next renderer step is a glyph atlas and batched text draw
path for comments.

## C ABI Foundation

`kuroko_capi` exposes an opaque `KurokoHandle` with create/destroy, media open, playback
commands, seek, state query, and event polling. The first event structure carries stable numeric
summaries for state, duration, position, track counts, buffering, video params, and errors. Detailed
track/string APIs can be layered on top without exposing Rust layouts.

For hosts that want Kuroko to own presentation, the C ABI also exposes `KurokoPresenterHandle`.
The presenter handle wraps the core player, Metal renderer, overlay timeline, and CoreAudio output.
The host still owns the platform view lifecycle and calls `kuroko_presenter_render_tick` from its
display timer, but it no longer needs to subscribe to frames or duplicate the demo render loop.

The C ABI accepts native Metal, wgpu, and Flutter Texture targets. That is enough for a Flutter host
to call the engine directly through `dart:ffi` once the Flutter plugin supplies per-platform native
view/surface glue, texture registration glue, and dynamic-library packaging. The C ABI is usable now,
but it is not yet a complete Flutter plugin by itself.

Texture and native view are different capability tiers. Flutter Texture is useful as a compatibility
path for SDR and platforms where native view composition is not ready. Native renderer-owned surfaces
remain the HDR/EDR path. On macOS this follows the earlier HDR finding: embedding through the
Flutter/AppKit view composition path can display but is prone to black flicker, so the validated HDR
path reserves a native layer region rather than sending frames through Flutter's compositor.

The macOS Flutter plugin should reuse the NipaPlay HDR PR shape: a window-hosted native overlay view,
transparent/cutout Flutter layout, attach retry, and surface generation checks so stale hide calls
from disposed Flutter widgets cannot hide a newly attached native video surface.

`docs/flutter_embedding.md` records the concrete plugin-facing flow and the capability split between
native presenter surfaces, Flutter Texture, and the future wgpu fallback.

## Metal Video Draw Milestone

`MetalRenderer::render_video_frame()` draws imported NV12/P010 planes into the attached
`CAMetalLayer` using a small Metal shader. RendererCore now describes the source/target color state,
render graph passes, tone mapping policy, and scaler policy in Rust before the Metal backend consumes
those parameters. The shader performs YCbCr sampling, transfer decode, a first Mobius/Reinhard/clip
tone-map path, and output encoding into `BGRA8Unorm`; this is an architectural seed, not the final
HDR/EDR renderer. Subtitle RGBA planes are blended in the same renderer-owned pass. The future HDR
compositor should switch the drawable and color pipeline to preserve PQ and EDR headroom, then add
native danmaku glyph batches on top.

Kuroko intentionally does not embed mpv gpu-next or libplacebo as the runtime renderer. Those projects
are useful design references for render graph shape, color management responsibilities, tone mapping,
scaling, and shader-chain ergonomics, but Kuroko's renderer is a clean Rust implementation with Metal
and future wgpu backends.

## WGPU Fallback Direction

The preferred Apple path stays platform-native Metal so HDR/EDR and VideoToolbox texture import can
use the best available APIs. For portability, `renderer::wgpu` defines a second renderer backend
boundary and WGPU surface handle model covering macOS/iOS views, Windows HWND, X11/Wayland, and
Android native windows. The current module validates lifecycle and API shape only; the next step is
to add the real wgpu dependency, surface creation, YCbCr/P010 upload or import strategy, and WGSL
shader path.
